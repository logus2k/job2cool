# job2cool — Project Plan

> HR assistant that turns a plain-language hiring need into a complete pack:
> job offer/description, technical interview, onboarding plan, team-culture
> hints, and more — generated by chat, grounded in company documentation
> (vector + graph RAG), and written live into an editable Markdown document.

_Status: drafting. Composed incrementally from the owner's ideas + a verified
investigation of `noted` (§9 is file:line-cited). No coding until §6 is settled._

---

## 1. Vision

An HR user receives a hiring need from any area of the company, in simple terms
(e.g. _"I need a software developer who knows Java and is comfortable working
with Agile teams"_). They paste/say it into the chat and, within moments, get a
**full hiring + interviewing + onboarding pack** — assembled by the Assistant
from company documentation and written, section by section, into a live Markdown
document they can read, edit, and export.

## 2. Scope & artifacts

One live Markdown document, assembled section by section:

1. **Job offer / description** — generated by **DPO-b01** from the plain need.
2. **Technical interview** — derived from the offer's required skills + seniority.
3. **Onboarding plan** — e.g. 30/60/90, grounded in company docs.
4. **Team-culture & integration hints** — grounded in the culture/values corpus.
5. _(extensible — more sections added the same way.)_

### Agent prompt(s) — first-class artifact (owner-flagged)

The **agent prompt** is what turns `gemma-4` into the HR orchestrator; it is
load-bearing for the whole flow (S5). Following the `cv_*` precedent, job2cool
adds its **own** agent_server presets (additive, non-destructive — same as cv
added `cv_assistant`/`cv_query_rewriter`/`cv_rag_judge`):
- **`job2cool_assistant`** — the main HR system prompt: persona; detect a
  job-offer need; **call `ma2-360m-dpo-b01`** for the draft, then refine + RAG-
  ground it; build vector/graph queries per section; compose **offer → technical
  interview → onboarding → culture**; **write each section into the live doc**
  via the doc tools; **cite** with the tag grammar cv/noted share.
- **`job2cool_rag_judge`** — Score panel judge (port of `cv_rag_judge`).
- _(maybe)_ **`job2cool_query_rewriter`** — focused retrieval-query rewriting.

Two layers, like cv-backend: a **static system prompt** (the preset) + **per-turn
directives** re-asserted by `job2cool-backend` (cv re-asserts citation/reasoning
rules every turn because small models are recency-biased; we mirror that). The
exact prompt text is an open authoring task (owner may provide a draft); it is
tightly coupled to §6.D (agentic vs pipeline) — the prompt encodes how much of
the sequence gemma drives vs the backend.

## 5. Preliminary flow _(open to discussion)_

1. User writes a need in plain terms (model = `gemma-4`, default).
2. Gemma detects a job-offer request and invokes **DPO-b01** to produce the offer.
3. Gemma analyses the offer and builds query expressions to search RAG
   (vector + graph) for each section's material.
4. Gemma composes a structured answer grounded in the docs and **writes each
   section into the open Markdown document**, which updates live.
5. User reads/edits/exports; citations trace every grounded claim to source.

**Insight from §9:** steps 1, 3, 4, 5 are *already implemented* in noted's
backend (native tool-calling loop, `search_docs` / `graph_and_vector_search` /
`research_topic`, the `create_doc`/`append_to_doc`/`replace_doc` buffer tools
that live-update the doc, and citation tags). The genuinely *new* work is step 2
(DPO-b01) plus the HR orchestration/skill that sequences the sections — and the
frontend graft.
